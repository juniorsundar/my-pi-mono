# r-plan produces a scoped starting point, not a fixed roadmap

r-plan (the new research planning subagent) decomposes the research question into areas, suggests initial search angles, and flags likely hard parts — but does not prescribe a full agent sequence. We chose this over a fixed roadmap (Option A: "r-search(query1) → r-learn(url2) → r-gap → r-synth") because a plan built before any search has happened is built on guesses. When iteration 3's search reveals something unexpected, a fixed roadmap either gets ignored (then why have it?) or followed rigidly (then the orchestrator can't adapt). Since r-gap already re-scopes mid-stream with actual evidence, a pre-research fixed plan is redundant with it. A scoped starting point gives the orchestrator a better frame for iteration 2 without fighting the adaptive loop.

Consequences: r-plan is always iteration 1 (hardcoded, not the orchestrator's choice), writes its output directly and immutably to a new `## Research Plan` section in state.md, and gets only `read`/`write` tools (no search — that blurs the line with r-search). The orchestrator uses the plan as a guide, not a script. Mid-research course corrections continue to go through `## Current Gaps` and `## Next Step` as they do today.
