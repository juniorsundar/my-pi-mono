# Map — `my-pi-mono`: consolidate extensions, migrate web-search to TypeScript

`wayfinder:map`

## Destination

A single git monorepo `github.com/juniorsundar/my-pi-mono`, loaded by pi as **one package** (one `pi.extensions` list, not separately-publishable sub-packages), containing every extension previously split across `~/.pi/agent/extensions/` and the `pi-subagents` repo — **with web-search migrated from Python to pure TypeScript**, so the package has one runtime, one language, and one test suite. Reached in two phases with no breakage: **phase 1** (DONE — #0002–#0007) built the monorepo, migrated the sources, cut over `settings.json`, and dropped the `spawnSubagent` event-bus handshake in favour of a direct barrel import. **Phase 2** migrates web-search's ~2.3k-line Python surface (`search.py`, `fetch.py`, `github.py`, `representation.py`, backed by `httpx` + `beautifulsoup4` + `lxml` + `readability-lxml` + `markdownify`) to TypeScript, deletes `scripts/`, `pyproject.toml`, `uv.lock`, and the `.venv`, and folds web-search's pytest suite into the one root vitest suite. The destination is reached when `my-pi-mono` is pure TS, no Python anywhere in the package, the four extensions all run from `my-pi-mono` alone, and the unified vitest suite passes.

### Phase 2 fidelity bar (locked this session)

**Shape + behavior-identical on the exposed surface; internals free reign.** The `web_search` and `web_fetch` tool input/output contracts and the `representation` module's exported `ExtractedDocument` shape must hold; the same *behaviours* must be preserved (SSRF private-IP blocking, truncation, content categorization, error paths, binary-download mode). Internals are free: a different readability library extracting slightly different prose is acceptable — the consumer is an LLM, not a diff. The TS test suite asserts shape + behaviour, not byte-identical prose. Reason-D composability (in-process import from `deep-research`/`mutation`) is satisfied by the port being *internally factored* (search/fetch/represent as separate functions so the seam is one `export` away) — **not** by exporting a speculative public internal module API before a real consumer exists.

## Notes

- **Domain:** pi coding-agent extensions. NodeNext/ESM TypeScript throughout. Today `web-search` is the sole non-TS surface — it shells out to Python via `uv` from a co-located `scripts/` dir + `.venv`; **phase 2 of this map removes that**, migrating web-search to TS so the whole package is one language, one runtime.
- **Reference pattern (3rd-party, not ours):** `juicesharp/rpiv-mono` — npm-workspaces monorepo of pi packages, root vitest + biome, `packages/<name>/` each with `package.json` `pi.extensions`. Use as a *guide* only; we are **one package with many extensions**, not separately-publishable sub-packages.
- **Skills every session should consult:** `/grilling` + `/domain-modeling` for any decision ticket; the pi extensions doc (`docs/extensions.md` in the pi install) and `docs/packages.md` for how pi discovers/loads packages.
- **Loading model:** `~/.pi/agent/extensions/` is auto-discovered; `settings.json` `packages` entries (`git:...`/`npm:...`) are installed. Today: the 4 in-scope extensions are auto-discovered from `extensions/`; `pi-subagents` is a `git:` package. After cutover: `my-pi-mono` is one `git:` package; `extensions/` no longer carries the moved ones.
- **Standing preference:** no breakage during migration — parallel-then-cut-over. Keep `extensions/` working until the monorepo builds + tests pass, then switch `settings.json` and remove the moved extensions.
- **Scope boundary:** `omniroute-pi-provider` is **out of scope** — it stays in `extensions/`, not in the monorepo.

## Decisions so far

- [web-search Python asset resolution under package loading](0003-websearch-asset-resolution.md) — pi loads extensions via `jiti` (runtime TS→CJS transform, not a bundler), so `__dirname` stays available and the committed `.venv` travels with a `git:` package; the web-search move is a verbatim copy, no `import.meta.url` rewrite or venv-rebuild needed.
- [Repo skeleton + single-package shape](0002-repo-skeleton.md) — flat `extensions/<name>/index.ts` layout; one root `package.json` with `pi.extensions: ["./extensions"]` (pi auto-discovers each subdir's index.ts); root tsconfig + vitest only, no biome/workspaces/per-package manifests; peer-deps `^0.80.0` (installed 0.80.10); repo `github.com/juniorsundar/my-pi-mono` cloned to `~/.pi/agent/git/github.com/juniorsundar/my-pi-mono`, loaded as `git:github.com/juniorsundar/my-pi-mono`.
- [Unify the `spawnSubagent` exporter: drop the event bus](0004-unify-spawn-exporter.md) — `extensions/subagents/index.ts` is the stable extension-facing barrel: it re-exports `spawnSubagent` and its real option/result types; deep-research imports only from that barrel, so the event-bus handshake and its load-order coupling are removed after the source move.
- [Migrate extension sources into `my-pi-mono`](0005-migrate-sources.md) — the local-only monorepo now has the locked one-package layout and parallel copies of all four extensions plus subagents (including web-search's trackable `.venv`); NodeNext imports and pi 0.80 typing are compatible, and root TS/Vitest plus Python tests pass.
- [Cutover: switch `settings.json` to `my-pi-mono`, remove moved extensions](0006-cutover.md) — phase 1 done: `my-pi-mono` is the single loaded package, the four in-scope extensions removed from `extensions/` (only `omniroute-pi-provider` remains), `pi-subagents` superseded; clean `pi --print` startup. Reverses #0003 on the venv: `.venv` stays gitignored, `uv run --project` auto-provisions it on first run (verified). #0004's implementation (direct-import refactor) graduates to [Drop the spawn event-bus handshake](0007-drop-spawn-handshake.md).
- [Drop the spawn event-bus handshake](0007-drop-spawn-handshake.md) — `deep-research` imports `spawnSubagent` + `SpawnSubagentResult` directly from the `extensions/subagents/index.ts` barrel (barrel now re-exports them); the `subagents:spawn:provide`/`:request` handshake, the structural mirror type, the mutable provider slot, and the subagents-side `:provide` emit are all removed. Scope correction (user-approved): #0005 was a verbatim copy and had not applied #0004's barrel change, so the barrel re-export + `:provide`-emit removal were folded into this ticket. `npm run typecheck` exit 0; 706 tests pass; `pi --print` clean. Phase 1 fully complete.
- [TS library stack for the web-search port](0008-websearch-ts-library-stack.md) — runtime: native `fetch` + `undici`, `@mozilla/readability` + `linkedom` (fallback `cheerio`), `turndown`, hand-rolled SSRF via `dns.promises.lookup` + `ipaddr.js`, no GitHub SDK; tests: `msw` (`setupServer` from `msw/node`). All deps lightweight, no jsdom transitives, ESM/jiti-clean. Highest risk: readability+linkedom compatibility — gate with a 10-line spike before committing. Research artifact: research/0008-ts-library-stack.md.
- [web-search port rewrite sequencing](0009-websearch-rewrite-sequencing.md) — parallel-then-cut: `scripts/*.py` stays the live source of truth (`index.ts` keeps shelling out to `uv`) while the TS port is built at `extensions/web-search/src/` module by module (search → representation → fetch → github); one final cut deletes all Python (`scripts/`, `pyproject.toml`, `uv.lock`, `.venv`, whole pytest suite) together once the TS port + vitest suite are green and `index.ts` switches to the `src/` barrel. Readability+linkedom spike gates the start (fall back to cheerio before `representation.ts` exists). Each module's pytest ported to its vitest twin alongside the TS twin; both runners run to the cut. Honours the standing no-breakage preference.
- [Shared test-utils for the web-search port](0010-websearch-shared-test-utils.md) — self-contained now, extract on second consumer: helpers live in `extensions/web-search/__tests__/helpers/` (inside `src`, sibling to `.test.ts`, reached by relative import; no vitest config change — the root `include` glob already covers it). No shared `test-utils/` built now (honours reason-D = B — no speculative shared surface with one consumer; today only web-search has HTTP-fetching tests). Lift the common bits (MSW `setupServer` factory, SSRF DNS-spy pattern, HTML fixture loader) into a root `test-utils/` exactly when a second extension adds its first HTTP-fetching test, not before.

- [Readability + linkedom spike: gate the web-search port start](0011-readability-linkedom-spike.md) — confirmed `@mozilla/readability` + `linkedom` interoperate under the root Node/Vitest setup; proceed with that primary HTML layer (Cheerio remains fallback only).
